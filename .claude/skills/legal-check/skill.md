# 法的リスクチェック

アーキテクチャとデータフローを起点に、アプリ全体の法的リスクを洗い出す。

判断記録は GitHub Issues に一元化する：

- **実施記録**: `legal-check-run` ラベルの Issue（1実行 = 1 Issue）。レポートと裁定の経緯を残す。open のまま = 未裁定の指摘がある状態。
- **受容済みリスク**: `legal-accepted` ラベルの closed Issue（1リスク = 1 Issue）。本文に受容理由と再検討の条件を記録する。

## 前提: このアプリのリスク許容度

- 個人開発・**非営利・無料**のアプリ。上場企業水準のコンプライアンス網羅を目指さない。
- 「法令・規約上の義務」と「あった方が望ましい」を峻別し、後者に対応は求めない。
- **新規指摘ゼロは失敗ではなく収束。** レポートの表を埋めるために指摘を作らない。

## 0. 判断記録の読み込み（必ず最初に行う）

```bash
# 受容済みリスク（本文の「再検討の条件」まで読む）
gh issue list --label legal-accepted --state all --json number,title,body

# 追跡中の既知リスク（open な Issue）
gh issue list --state open --json number,title

# 前回の実施記録（裁定が終わっているか確認）
gh issue list --label legal-check-run --state all --limit 3 --json number,title,state
```

あわせて `docs/SCRAPING_POLICY.md` を読む（スクレイピング関連の判断はここで確定済み）。

**ルール: 以下は再指摘しない。**

- `legal-accepted` の項目（本文の「再検討の条件」に該当する変化があった場合を除く）
- open な Issue で既知のリスク（Issue 番号を添えて言及するのは可）
- 表現を変えただけの同種の指摘（例: 受容済みの文言・命名について別の改善案を出す）

前回の実施 Issue が open のままなら、未裁定の指摘が残っている。新規チェックの前にユーザーに知らせる。

## 1. 全体像の把握

以下を読み、アプリのデータフロー全体を把握する：

- `docs/ARCHITECTURE.md` - アーキテクチャ・構成図
- `docs/DATABASE_DESIGN.md` - DB 設計（存在する場合）

特に以下の観点でデータの流れを整理する：
- **外部から取得するデータ**: 何を、どこから、どうやって取得しているか
- **外部に送信するデータ**: 何を、どのサービスに送っているか
- **ユーザーから収集するデータ**: 何を、どこに保存しているか

## 2. DB スキーマの確認

`supabase/migrations/` の主要テーブル定義を確認し、以下をリストアップする：

- 個人情報に該当するカラム（ユーザーID、表示名、メール等）
- 外部コンテンツに由来するカラム（URL、タイトル、画像等）
- RLS（Row Level Security）の設定状況

## 3. 外部連携のチェック

ステップ1で特定したデータフローごとに、関連コードを読んで確認する：

### 外部データ取得（スクレイピング等）
- 取得データの範囲（メタデータのみ or 本文含む）
- `docs/SCRAPING_POLICY.md` と実装が一致しているか
- robots.txt・対象サイト利用規約は SCRAPING_POLICY.md の確認記録を正とする。**記載の確認時点から6ヶ月以上経過している場合のみ** WebFetch で再確認し、SCRAPING_POLICY.md の確認時点を更新する

### 外部サービスへのデータ送信（AI API 等）
- 送信データの内容と範囲
- 個人情報や第三者の著作物が含まれないか

### プラットフォーム連携（LINE 等）
- ユーザーデータの取得・保存範囲
- プラットフォームの開発ガイドラインとの整合性
- 退会・データ削除フローの有無

### 外部コンテンツの表示
- 表示方法（直リンク / プロキシ / コピー保存）
- 著作物の複製に該当しないか

## 4. 法的ページの確認

`src/components/features/legal/` のプライバシーポリシー・利用規約を読み、以下を確認：

- ステップ3で特定した全データフローがポリシーに記載されているか
- データ保持期間、削除方法が明記されているか
- 外部サービスの利用がすべて記載されているか

## 5. 重要度の判定基準（線引き）

### 要対応（Issue 化を提案する）— 以下のいずれかに該当する場合のみ

- **法令上の義務違反が具体的に説明できる**（例: 同意なく個人情報を第三者提供している、著作物の本文・画像を複製・公衆送信している、法定の表示義務を欠く）
- **公表しているポリシー・規約と実装が矛盾している**（ユーザーに事実と異なる説明をしている状態）
- **プラットフォーム・対象サイト規約への明確な違反**で、アカウント停止や取得ブロックの現実的リスクがある

### 受容（記録用 Issue を作るだけ。対応しない）

- ベストプラクティス・業界慣行レベルの改善（文言の丁寧さ、命名、体裁）
- 条件が変わったら初めて問題になる項目（例:「営利化したら○○」）→ **発動条件つき**で記録
- 理論上のリスクで、実害に至る具体的な経路が説明できないもの

**判断に迷ったら「受容」に倒し、「要確認」として人間に委ねる。** 要対応に入れてよいのは、上の基準のどれに該当するかを一文で言えるものだけ。

## 6. 結果レポートと実施 Issue の作成

レポートをユーザーに提示し、同じ内容で実施 Issue を作成する：

```bash
gh issue create --title "法的リスクチェック YYYY-MM-DD" --label legal-check-run --body "<レポート>"
```

レポート形式（新規指摘がない場合は「要対応」以降の表は作らない）：

```
## 法的リスクチェック結果（YYYY-MM-DD）

対象コミット: <HEAD の hash>

### データフロー概要
[ステップ1で整理した外部連携の一覧表]

### 要対応
| 内容 | 該当する判定基準 | 推奨対応 |
|------|------------------|----------|
| ... | 法令義務 / ポリシー矛盾 / 規約違反 | ... |

### 受容（記録のみ・対応不要）
- [項目]（再検討の条件: ...）

### 要確認（人間の判断が必要）
- [判断が必要な項目]

**注意:** これは技術的観点からのチェックであり、法的助言ではありません。
```

**新規指摘がない場合**は「全データフローを確認。新規指摘なし。」というレポートで実施 Issue を作成し、裁定は不要なので**即 close する**（実施記録として残す）。

## 7. 裁定の記録

ユーザーと対応方針を決めたら、裁定ごとに記録する：

- **要対応** → 通常の Issue を作成し、実施 Issue にリンクをコメント
- **受容** → まず既存の `legal-accepted` Issue に同種・同カテゴリのものがないか確認する。**あれば新規作成せず、その Issue にコメントで追記する**（受容 Issue はカテゴリ単位でまとめ、無駄に増やさない）。なければ記録用 Issue を作成して**即 close** する：

```bash
gh issue create --title "[受容] <リスクの要約>" --label legal-accepted --body "<下記フォーマット>"
gh issue close <番号> --reason "not planned"
```

記録用 Issue の本文フォーマット：

```
## 指摘内容
[何がリスクとされたか]

## 受容理由
[なぜ対応しないと判断したか]

## 再検討の条件
[この条件に該当したら再検討する。/legal-check はこの条件に該当しない限り再指摘しない]

記録元: #<実施 Issue 番号>
```

**すべての指摘の裁定が済んだら実施 Issue を close する。** 裁定が持ち越しになる場合は open のままにし、残項目をコメントに残す。
